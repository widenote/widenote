当前结构问题

从产品心智看
首页入口太集中

Recap、Timeline、Search、Memory、Settings 都挤在标题右侧。 用户需要猜每个图标代表什么，首页像控制台而不是每日返回面。

核心循环被拆散

Timeline、Memory、Recap 都是核心体验，但不是一级导航。 它们可用，却没有形成一条稳定的用户路径。

插件页承担过多设置

Packs、权限、模型、备份、Trace 都在插件页出现。 对开发者友好，但会削弱“插件市场 / 能力管理”的清晰心智。

设计目标

布局应服务什么
  • 首页第一眼回答“今天发生了什么”和“我如何继续记录”。
  • 记录入口必须够快，但不占掉所有内容阅读空间。
  • Timeline 是完整回看，不一定要抢首页主体。
  • Memory 是可纠错的长期上下文，不应该变成每日确认任务。
  • 插件页要么是市场，要么是高级控制台，不能两种心智混在一起。
  • 设置、备份、模型、Trace 更适合归入右上角设置或高级区。

首首页 聊对话 +记录 办待办 插插件

记录 Sheet 覆盖在当前上下文上，用户不离开首页。

1. 进入首页

默认看到今日回顾草稿、最近两三条记录，以及明显但不喧宾夺主的记录入口。

2. 打开记录 Sheet

底部中间主按钮或首页“继续记录”打开 Sheet。文字、图片、相册、文件入口同层。

3. 保存原始记录

原始记录立即落本地；AI 输出只作为派生对象，不覆盖输入。

4. 轻量反馈

首页显示“已保存，可查看记录”，但不把 Memory 确认、Todo 确认塞回首页。

5. 回看与纠错

用户从“记录”页看完整流，从详情页看来源、派生卡、Memory、Todo、Trace。

后台录音

后台录音不放进 Sheet 里当一个普通按钮，而是首页/底部的独立任务。 录音中显示浮动状态，停止后打开 Sheet 让用户补充文字上下文。

录音中 停止后补充说明 作为源材料保存
Memory 处理

普通 Memory 自动接受。只有敏感、低置信、冲突、策略不清的内容进入复核。 首页只提示“有待复核”，不默认展示确认列表。
