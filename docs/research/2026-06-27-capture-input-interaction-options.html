WideNote 首页输入交互方案

目标先收窄到一个问题：把“快速记录”从调试控制台改成明确、低摩擦、不会被误读成录音机的输入入口。

设计决策稿 · 2026-06-27

当前实现诊断

首页像运行控制台

第一屏后面连续展示处理阶段、卡片、洞察、记录、Memory、Trace。它们都重要，但不应该和输入入口抢第一眼注意力。

快速记录内重复暴露能力

文字/语音/媒体分段、模式说明面板、底部相机/相册/语音/记录按钮同时出现，用户需要先学习结构再输入。

提交动作被录音语义污染

现在主按钮是实心录制圆点加“记录”。在语音场景里，这会自然读成开始录音，而不是保存这条原始记录。

Memory / 待办不应抢首页

Memory 候选可以是后台策略对象，但默认捕获应静默进入。待办已经有专门 tab，首页只需要提示入口或状态，不放任务列表。

竞品输入模式

Memex：单输入流

主心智是“捕获任何碎片”：文字、照片、长按录音被收进同一个输入流程，用户不先面对一排同权按钮。

Omi / Limitless：录音是主任务

它们的产品承诺是录会议和对话，录音入口大而明确是合理的。WideNote 若不是 voice-first，就不该让录音图标统治提交动作。

Supermemory / Granola：入口分层

Supermemory 用 Add Memory 对话框区分 Note、Link、File、Connect；Granola 把一键捕获和手写 note editor 分开，减少首页按钮密度。

场景重新梳理

场景 1：沉浸式输入

用户主动写一段东西，可能很长，也可能附带图片、截图、链接或文件。入口应是“新记录”，进入大弹窗后专心输入和保存。

场景 2：后台录音

用户开始录音后继续做别的事。首页只显示持续录音状态、计时、暂停/停止；停止后音频成为一条可保存或补充说明的原始记录。

首页职责

首页负责“开始记录”和“看最近发生了什么”。Memory 管理、待办处理、Trace 调试和候选复核应下沉到对应 tab 或二级入口。
